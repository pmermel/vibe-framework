# vibe.yaml Schema Reference

Every project generated by vibe-framework must include a root `vibe.yaml`. This file is the single source of truth for project configuration consumed by both providers, bootstrap automation, and deployment workflows.

## Required Fields

### `name`
Project name. Used as a prefix for Azure resource names and GitHub repo name.
- Type: string
- Pattern: lowercase letters, numbers, hyphens only
- Example: `my-app`

### `template`
Project scaffold template.
- Type: string
- Allowed values: `nextjs`, `react-vite`, `node-api`
- Example: `nextjs`

### `adapter`
Deployment adapter.
- Type: string
- Allowed values: `container-app` (default), `static-web-app`
- Example: `container-app`

### `providers`
List of AI providers authorized to work on this project.
- Type: array of strings
- Allowed values: `claude`, `codex`
- Example: `[claude, codex]`

### `branch_policy`
Branch naming and protection rules.

| Field | Type | Description |
|---|---|---|
| `feature_prefix` | string | Prefix for feature branches. Default: `feature/` |
| `staging` | string | Staging branch name. Default: `develop` |
| `production` | string | Production branch name. Default: `main` |

### `build`
Build commands run by GitHub Actions workflows.

| Field | Type | Description |
|---|---|---|
| `install` | string | Dependency install command |
| `test` | string | Test command |
| `build` | string | Build command |
| `output` | string | Build output directory |

### `deploy`
Deployment targets per environment.

```yaml
deploy:
  preview:
    target: container-app   # or static-web-app
  staging:
    target: container-app
  production:
    target: container-app
```

### `azure`
Azure resource configuration.

| Field | Type | Description |
|---|---|---|
| `region` | string | Azure region. Default: `eastus2` |
| `resource_group` | string | Resource group name |
| `registry` | string | Azure Container Registry name ‚Äî alphanumeric only, globally unique, 5‚Äì50 chars. Convention: app name with hyphens removed + `acr` suffix (e.g. `myappacr`) |
| `container_app_environment` | string | Container Apps environment name |
| `preview_app_prefix` | string | Prefix for ephemeral preview Container Apps |
| `staging_app` | string | Staging Container App name |
| `production_app` | string | Production Container App name |

### `github`
GitHub repository and workflow configuration.

| Field | Type | Description |
|---|---|---|
| `repo` | string | GitHub repo in `owner/repo` format |
| `issues_as_work_queue` | boolean | Use GitHub Issues as the task queue |
| `workflow_refs` | object | Pinned reusable workflow references |

#### `workflow_refs`
This is the canonical source of truth for reusable workflow version pinning. Bootstrap and upgrade automation generates thin wrapper workflow files from these values.

```yaml
github:
  workflow_refs:
    preview: pmermel/vibe-framework/.github/workflows/reusable-preview.yml@v1
    staging: pmermel/vibe-framework/.github/workflows/reusable-staging.yml@v1
    production: pmermel/vibe-framework/.github/workflows/reusable-production.yml@v1
```

- References must be pinned to a release tag or commit SHA ‚Äî never `@main`.
- Updating these refs and re-running bootstrap generates updated wrapper workflows.

### `approvers`
GitHub usernames that may approve production deployments.
- Type: array of strings
- Example: `[your-github-username]`

## Provider Neutrality Rules

- No provider-specific fields allowed in `vibe.yaml`.
- No Anthropic, OpenAI, Claude, or Codex references in field names or values.
- `providers` list describes which providers are authorized, not how they work.

## Full Example

```yaml
name: my-app
template: nextjs
adapter: container-app
providers:
  - claude
  - codex

branch_policy:
  feature_prefix: feature/
  staging: develop
  production: main

build:
  install: npm ci
  test: npm test
  build: npm run build
  output: .next

deploy:
  preview:
    target: container-app
  staging:
    target: container-app
  production:
    target: container-app

azure:
  region: eastus2
  resource_group: my-app-rg
  registry: myappacr
  container_app_environment: my-app-env
  preview_app_prefix: my-app-pr
  staging_app: my-app-staging
  production_app: my-app-prod

github:
  repo: YOUR_GITHUB_USERNAME/my-app
  issues_as_work_queue: true
  workflow_refs:
    preview: YOUR_GITHUB_USERNAME/vibe-framework/.github/workflows/reusable-preview.yml@v1
    staging: YOUR_GITHUB_USERNAME/vibe-framework/.github/workflows/reusable-staging.yml@v1
    production: YOUR_GITHUB_USERNAME/vibe-framework/.github/workflows/reusable-production.yml@v1

approvers:
  - YOUR_GITHUB_USERNAME
```
