# Deployment Contract

Authoritative reference for GitHub environment names, required secrets and variables, ACR pull-auth model, managed identity requirements, layer ownership, and failure/rollback behavior. Applies to all projects generated by vibe-framework using the `container-app` adapter.

---

## Provider Neutrality

This contract is provider-neutral. Bootstrap automation, Bicep templates, and GitHub Actions workflows enforce it regardless of whether Claude Code or Codex triggered the underlying work. No provider-specific logic appears in any deployment path.

---

## GitHub Environment Names

Three canonical environment names are used across all generated projects. These names are fixed — do not rename them. Bootstrap automation creates them in the project repo by name, and the reusable workflows reference them by name for OIDC trust scoping and approval gates.

| Environment | Purpose | Branch trigger | Approval gate |
|---|---|---|---|
| `preview` | Ephemeral per-PR environment, one per open PR | PR open/push/close | None — deploys automatically |
| `staging` | Long-lived environment tracking `develop` (or `vibe.yaml branch_policy.staging`) | Push to staging branch | None — deploys automatically |
| `production` | Long-lived environment tracking `main` (or `vibe.yaml branch_policy.production`) | Push to production branch | Required — reviewer from `vibe.yaml approvers` |

The `preview` GitHub environment exists for OIDC trust scoping only — it does not pause the workflow. The approval gate exists only on `production`.

## Required GitHub Actions Secrets

All three secrets are set by bootstrap automation on each of the three GitHub environments (`preview`, `staging`, `production`). They must be environment-scoped — not stored as repository-level secrets — so that OIDC subjects are correctly scoped per environment.

| Secret | What it is | Set by | Used by |
|---|---|---|---|
| `AZURE_CLIENT_ID` | Client (application) ID of the Azure AD service principal for this environment — output of `oidc-federated-credential.bicep` for the matching `githubEnvironment` param | Bootstrap automation (`configure_repo`) after `configure_cloud` runs Bicep | All four reusable workflows |
| `AZURE_TENANT_ID` | Azure AD tenant ID for the subscription | Bootstrap automation (`configure_repo`) — same value on all three environments | All four reusable workflows |
| `AZURE_SUBSCRIPTION_ID` | Azure subscription ID where the project's resource group lives | Bootstrap automation (`configure_repo`) — same value on all three environments | All four reusable workflows |

Each environment has its own `AZURE_CLIENT_ID` because each environment has its own service principal and federated credential with a distinct OIDC subject. `AZURE_TENANT_ID` and `AZURE_SUBSCRIPTION_ID` are shared values but must still be set on each environment so that `secrets: inherit` works correctly within the environment context.

No long-lived credentials (passwords or client secrets) are stored anywhere. GitHub Actions runners exchange OIDC tokens for short-lived Azure access tokens through the Azure AD federated credential.

## OIDC Federated Credential Subjects

Federated credentials are provisioned by `infrastructure/oidc-federated-credential.bicep`, deployed once per environment per project during `configure_cloud`.

| GitHub Environment | OIDC subject |
|---|---|
| `preview` | `repo:<owner>/<repo>:environment:preview` |
| `staging` | `repo:<owner>/<repo>:environment:staging` |
| `production` | `repo:<owner>/<repo>:environment:production` |

The federated credential issuer is always `https://token.actions.githubusercontent.com` with audience `api://AzureADTokenExchange`.

---

## Azure RBAC Requirements

Each service principal (one per environment) receives the following roles, granted by `infrastructure/oidc-federated-credential.bicep`:

| Role | Scope | Purpose |
|---|---|---|
| `Contributor` | Project resource group | Create, update, and delete Container Apps and related resources |
| `AcrPush` | Project Azure Container Registry | Push container images via `az acr build` |

The `AcrPull` role for Container Apps is granted separately — see ACR Pull Auth Model below.

## Bicep Parameters: `oidc-federated-credential.bicep`

Called once per environment per project. The following parameters are required:

| Parameter | Type | Description |
|---|---|---|
| `appName` | string | Short project name — used to label the app registration in Azure AD |
| `githubOrg` | string | GitHub organisation or user owning the repository |
| `githubRepo` | string | GitHub repository name (without org prefix) |
| `githubEnvironment` | string (enum) | `preview`, `staging`, or `production` |
| `resourceGroupId` | string | Resource ID of the project resource group |
| `registryId` | string | Resource ID of the project ACR |

Outputs:

| Output | Description | Stored as |
|---|---|---|
| `clientId` | Azure AD application (client) ID | `AZURE_CLIENT_ID` on the matching GitHub environment |
| `tenantId` | Azure AD tenant ID | `AZURE_TENANT_ID` on all three GitHub environments |
| `principalId` | Service principal object ID | Used internally by bootstrap for verification |

## Bicep Parameters: `container-apps-env.bicep`

Called once per project during `configure_cloud`. Provisions the dedicated Container Apps environment, ACR, staging Container App, and production Container App.

| Parameter | Type | Description | Source in vibe.yaml |
|---|---|---|---|
| `appName` | string | Project name prefix for all resource names | `name` |
| `region` | string | Azure region | `azure.region` |
| `registryName` | string | ACR name — alphanumeric, globally unique, 5–50 chars | `azure.registry` |
| `stagingAppName` | string | Full staging Container App resource name | `azure.staging_app` |
| `productionAppName` | string | Full production Container App resource name | `azure.production_app` |

The Container Apps environment is named `${appName}-env` (matching `azure.container_app_environment` in vibe.yaml). If a different name is needed, pass a separate param — the current Bicep derives the environment name from `appName` using the same convention.

Outputs used by bootstrap:

| Output | Used for |
|---|---|
| `acrLoginServer` | Stored as reference; passed to workflows |
| `acrId` | Passed to `oidc-federated-credential.bicep` as `registryId` |
| `stagingFqdn` | Posted to PR or stored as variable |
| `productionFqdn` | Posted to PR or stored as variable |
| `stagingPrincipalId` | Verification — confirms managed identity exists |
| `productionPrincipalId` | Verification — confirms managed identity exists |

## ACR Pull Auth Model

All Container Apps pull images from a private Azure Container Registry using system-assigned managed identity with the `AcrPull` role. No admin credentials or stored registry passwords are used.

### Staging and Production — Provisioned at Bootstrap

Managed identity and AcrPull are assigned by `infrastructure/container-apps-env.bicep` at project creation time:

- `stagingApp` and `productionApp` resources in Bicep declare `identity: { type: 'SystemAssigned' }`.
- Role assignments `stagingAcrPull` and `productionAcrPull` in Bicep grant `AcrPull` (role ID `7f951dda-4ed3-4680-a7ca-43fe172d538d`) on the ACR to each app's managed identity principal.
- The Bicep role assignment names are deterministic GUIDs derived from ACR ID + app ID + role ID, making them idempotent on re-deployment.
- Because identity and AcrPull are in place before the first workflow run, `reusable-staging.yml` and `reusable-production.yml` call `az containerapp update` directly — no runtime identity wiring.

### Preview — Wired at First Deploy

Preview Container Apps are created dynamically by `reusable-preview.yml` because they do not exist until a PR is opened. The first deploy for each PR follows this bootstrap sequence to resolve the identity chicken-and-egg problem (identity principal ID is unknown until the app exists, but AcrPull must reference the principal ID):

1. **Create** the Container App with a public placeholder image (`mcr.microsoft.com/azuredocs/containerapps-helloworld:latest`) and `--assign-identity "[system]"`.
2. **Retrieve** the system identity's principal ID from the created app.
3. **Assign** `AcrPull` on the ACR to that principal ID via `az role assignment create`.
4. **Wait** 60 seconds for Azure IAM role propagation (global propagation can take up to 2 minutes; 60 seconds is the practical minimum for this workflow step).
5. **Update** the Container App to the private ACR image using `--registry-identity system`.

### Preview — Subsequent Pushes

System identity and AcrPull are already in place from step 3 above. The workflow calls `az containerapp update` with the new image — no identity steps.

### Preview — Cleanup

On PR close, the preview workflow deletes both the Container App (removing the managed identity) and the `pr-<N>` ACR image. No explicit role assignment cleanup is needed because the role assignment is scoped to the now-deleted principal.

## Bootstrap Action Responsibilities

### `configure_cloud`

Runs `infrastructure/oidc-federated-credential.bicep` three times (once per environment: `preview`, `staging`, `production`) and runs `infrastructure/container-apps-env.bicep` once. After completion:

- Project resource group exists with staging and production Container Apps provisioned.
- Three Azure AD app registrations exist, each with a federated credential scoped to one GitHub environment.
- RBAC roles are assigned.
- Bicep outputs (`clientId`, `tenantId`, ACR resource ID) are returned to the caller for use by `configure_repo`.

### `configure_repo`

Receives Bicep outputs from `configure_cloud` and:

- Creates the three GitHub environments (`preview`, `staging`, `production`) in the project repo via the GitHub App.
- Sets `AZURE_CLIENT_ID` as an environment secret on each environment using the per-environment `clientId` from Bicep output.
- Sets `AZURE_TENANT_ID` and `AZURE_SUBSCRIPTION_ID` as environment secrets on all three environments.
- Configures required reviewers on the `production` environment using the `approvers` list from `vibe.yaml`.
- Applies branch protection rules per `vibe.yaml branch_policy`.

Neither `configure_cloud` nor `configure_repo` deploys application code or triggers workflows. Deployment is always initiated by GitHub Actions reacting to branch events.

## Preview Concurrency and TTL Controls

Cost-safety defaults come from `vibe.yaml` and are enforced by workflow and bootstrap automation:

| vibe.yaml field | Default | Enforcement point |
|---|---|---|
| `deploy.preview.max_concurrent` | `3` | Enforced in `reusable-preview.yml` on first deploy for a PR. Before creating a new preview app, the workflow counts active apps by prefix and deletes the oldest (app + ACR image) if the limit is reached. Update pushes to an existing PR skip enforcement. |
| `deploy.preview.ttl_hours` | `48` | Converted to `max_age_days` (ceiling division by 24) for the `reusable-preview-ttl-cleanup.yml` wrapper input. Primary cleanup is still PR-close; TTL cleanup is the backstop for abandoned previews. |

## Failure Behavior

### Preview Deploy Failure

- The deploy job fails and the PR check reports failure.
- If the Container App was not yet created when the failure occurred, no Azure resource is left running.
- If the failure occurred after Container App creation, the app may be left in a partially-wired state (placeholder image, no AcrPull yet). Re-pushing to the PR reruns the workflow from the top — the workflow detects the app already exists (update path) and skips the creation steps, which may cause the update to fail if AcrPull was not yet assigned. In that case, manually delete the partial Container App from Azure and re-push.
- There is no automatic retry.

### Staging Deploy Failure

- The deploy job in `reusable-staging.yml` fails and GitHub reports a failed check on the commit.
- `az containerapp update` is transactional from the Container Apps perspective: if the update step fails, the Container App continues running the previous revision.
- No automatic revert. Fix the cause, push to `develop`, and the workflow reruns.

### Production Deploy Failure (after approval)

- The deploy job in `reusable-production.yml` fails after the approval gate is passed.
- Azure Container Apps keeps the previous revision running. Production is not left in a broken state.
- No automatic rollback. A new push to `main` (which requires a new approval) re-attempts deployment.

## Azure Container Apps Revision Model and Rollback

Azure Container Apps creates a new revision on every `az containerapp update`. vibe-framework uses single-revision mode (100% traffic to the new revision) by default.

- The previous revision is deactivated but not deleted immediately.
- If the new revision fails platform-level health checks, Container Apps may route traffic back to the previous revision. This is a platform behavior, not a guaranteed framework feature.

### Rollback in v1

There is no automatic rollback in v1. This is an explicit decision.

Manual rollback options:

1. Re-run the previous successful workflow run from the GitHub Actions UI — this rebuilds and redeploys from the same commit.
2. Revert the offending commit, push to the branch, and let the workflow run normally.

Auto-rollback is deferred to a future version. Reliable auto-rollback requires application-level health checks with a clear success signal, which are project-specific and not yet part of the framework contract. Container Apps retains previous revisions, so the rollback path is always available.

## Static Web App Adapter

When `vibe.yaml adapter: static-web-app` is set, the deployment model is different:

- Azure Static Web Apps is used instead of Container Apps.
- The ACR pull-auth model described in this document does not apply (no container image; Azure SWA handles deployment directly from a build artifact).
- GitHub environments (`preview`, `staging`, `production`) still apply, with the same names and approval rules.
- `infrastructure/static-web-app.bicep` replaces `container-apps-env.bicep` for provisioning.
- A separate reusable workflow set (not yet implemented in v1) handles static deployment.

This document covers only the `container-app` adapter. Static adapter contracts will be documented here when the static workflow set is implemented.
