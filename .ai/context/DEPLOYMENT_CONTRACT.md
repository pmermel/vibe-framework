# Deployment Contract

Authoritative reference for GitHub environment names, required secrets, ACR pull auth model, failure behavior, and rollback expectations. Applies to all projects generated by vibe-framework using the `container-app` adapter.

---

## GitHub Environment Names

| Environment | Purpose | Approval Gate |
|---|---|---|
| `preview` | Ephemeral per-PR environment | None — deploys automatically on PR push |
| `staging` | Long-lived environment tracking `develop` | None — deploys automatically on merge to `develop` |
| `production` | Long-lived environment tracking `main` | Required — must be approved by a reviewer in the `approvers` list |

These names are canonical. Bootstrap automation (`configure_repo`) creates these three environments in the generated project's GitHub repository. Do not rename them.

The `preview` environment in GitHub exists for OIDC trust scoping only — it does NOT have an approval gate configured.

---

## Required GitHub Actions Secrets

All three secrets are set on each GitHub environment (`preview`, `staging`, `production`) by bootstrap automation. They must not be stored as repository-level secrets.

| Secret | What It Is | Who Sets It | Which Workflows Need It |
|---|---|---|---|
| `AZURE_CLIENT_ID` | Client (application) ID of the Azure AD service principal used for OIDC login | Bootstrap automation (`configure_repo`) after `configure_cloud` creates the service principal | All four reusable workflows |
| `AZURE_TENANT_ID` | Azure AD tenant ID for the subscription hosting the project's resources | Bootstrap automation (`configure_repo`) | All four reusable workflows |
| `AZURE_SUBSCRIPTION_ID` | Azure subscription ID where the project's resource group lives | Bootstrap automation (`configure_repo`) | All four reusable workflows |

No long-lived credentials (passwords, client secrets) are stored. Authentication uses GitHub Actions OIDC: the runner exchanges its OIDC token for a short-lived Azure access token via the federated credential on the service principal.

OIDC federated credentials are scoped per environment:

| GitHub Environment | OIDC Subject |
|---|---|
| `preview` | `repo:<owner>/<repo>:environment:preview` |
| `staging` | `repo:<owner>/<repo>:environment:staging` |
| `production` | `repo:<owner>/<repo>:environment:production` |

## ACR Pull Auth Model

All Container Apps pull images from a private Azure Container Registry using system-assigned managed identity with the AcrPull role. No admin credentials are used.

### Staging and Production

- Managed identity is assigned at provision time by `infrastructure/container-apps-env.bicep`.
- AcrPull role on the registry is granted by Bicep during bootstrap.
- Reusable workflows call `az containerapp update` directly — no identity wiring at deploy time.

### Preview (first deploy for a PR)

Preview Container Apps are created dynamically by the workflow. Because the Container App must exist before its identity's principal ID is known, creation follows a bootstrap sequence:

1. Create the Container App with a public placeholder image and `--assign-identity "[system]"`.
2. Retrieve the system identity's principal ID from the created app.
3. Assign `AcrPull` on the ACR to that principal ID.
4. Wait 60 seconds for Azure IAM role propagation.
5. Update the Container App to the private ACR image using `--registry-identity system`.

### Preview (subsequent pushes to the same PR)

System identity and AcrPull are already in place from the first deploy. The workflow calls `az containerapp update` with the new image tag — no identity steps needed.

### Layer Ownership

| Contract Element | Owner |
|---|---|
| Managed identity assignment (staging/production) | Bicep (`infrastructure/container-apps-env.bicep`) |
| AcrPull role grant (staging/production) | Bicep |
| Managed identity assignment (preview, first deploy) | `reusable-preview.yml` at runtime |
| AcrPull role grant (preview, first deploy) | `reusable-preview.yml` at runtime |
| OIDC federated credentials | Bicep (`infrastructure/oidc-federation.bicep`) via `configure_cloud` |
| GitHub environment secrets | Bootstrap automation (`configure_repo`) |
| GitHub environment approval gate (production only) | Bootstrap automation (`configure_repo`) using `approvers` from `vibe.yaml` |

## Failure Behavior

### Staging Deploy Failure

- The `deploy` job in `reusable-staging.yml` fails and the workflow reports a failed check on the commit.
- `az containerapp update` is atomic from the Container Apps perspective: if the update step fails, the Container App continues running the previous revision.
- No automatic revert or retry. The previous revision stays live.
- Fix the issue, push to `develop`, and the workflow re-runs.

### Production Deploy Failure (after approval)

- The `deploy` job in `reusable-production.yml` fails after the approval gate has been passed.
- Azure Container Apps keeps the previous revision running. The production environment is not left in a broken state.
- No automatic rollback. The previous revision remains live until a successful deploy replaces it.
- A new push to `main` (requiring a new approval) is the mechanism to re-attempt deployment.

## Azure Container Apps Revision Model and Rollback

Azure Container Apps creates a new revision on every `az containerapp update`. The previous revision is not deleted automatically.

In the default single-revision mode used by vibe-framework:

- Traffic is shifted 100% to the new revision on update.
- The old revision is deactivated but not immediately deleted.
- If the new revision fails health checks, Container Apps may route traffic back to the previous revision (platform behavior — not guaranteed in all configurations).

### Rollback in v1

There is no automatic rollback in v1. This is an explicit decision.

To roll back manually:

1. Re-run the previous successful workflow run via the GitHub Actions UI, **or**
2. Update the image tag in the workflow inputs to a known-good tag and push to trigger a new run.

Auto-rollback is deferred to a future version when observability tooling is in place to define a reliable success signal.

### Why No Auto-Rollback in v1

- Defining "deployment success" reliably requires application-level health checks, which are project-specific.
- Auto-rollback without a clear success signal risks flapping between revisions.
- Manual rollback via GitHub Actions UI is explicit, auditable, and sufficient for v1 use cases.
- Container Apps retains previous revisions, so the rollback path is always available without extra state management.
